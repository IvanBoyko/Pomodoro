### 3. Silent error swallowing in `TimerViewModel` (multiple locations)

**Importance:** Medium | **Urgency:** Soon | **Cost:** Small

The codebase uses `try?` / `catch {}` without any logging in several critical paths. `CLAUDE.md` explicitly warns against this pattern: "Silent `try?` / `catch {}` swallow exactly the errors that take a device cycle to surface."

Locations:

| File | Line context | What's swallowed |
|------|-------------|------------------|
| `TimerViewModel.swift` | `try? AlarmManager.shared.pause(id:)` | Pause failure |
| `TimerViewModel.swift` | `try? AlarmManager.shared.resume(id:)` | Resume failure |
| `TimerViewModel.swift` | `try? AlarmManager.shared.cancel(id:)` | Cancel failure (×3 locations) |
| `TimerViewModel.swift` | `catch {}` in `startLiveActivity()` | Live Activity start failure |
| `TimerViewModel.swift` | `catch {}` in `triggerCompletionHaptic()` | Haptic pattern failure |
| `TimerViewModel.swift` | `catch {}` in `scheduleAlarmKitTimer()` | AlarmKit schedule failure (has a `print` for this one) |

The AlarmKit schedule path already has a `print` on failure — the other paths should follow the same pattern. At minimum, a `print("AlarmKit pause failed: \(error)")` costs nothing and saves a device debugging cycle.

- **Action:** Add `print` with the error to each empty `catch {}` / `try?` in TimerViewModel.

### 4. `Color(hex:)` produces silent black for invalid input

**Importance:** Medium | **Urgency:** Soon | **Cost:** Small

`ColorExtension.swift` — `Scanner.scanHexInt64` returns `false` for empty or malformed strings, but `rgbValue` remains the uninitialized value (`0`) and the Color is constructed from all-zero components. This silently produces black instead of flagging the error.

In practice the input always comes from `Color.categoryColors` (a hardcoded valid list) or `PomodoroCategory.colorHex` which is set from the same palette, so this won't fire today. But `colorHex` is user-editable via the Categories form — nothing prevents a future code path from feeding garbage in.

- **Action:** Guard on `scanner.scanHexInt64(&rgbValue)`; return a fallback (e.g., `.gray`) or `fatalError` in debug if parsing fails.

### 5. `CategoryPickerSheet` can dead-end the UI if TimerViewModel state is nil

**Importance:** High | **Urgency:** Soon | **Cost:** Small

`CategoryPickerSheet.saveSession(category:)` guards on `timerVM.completedStartDate` and `timerVM.completedEndDate`. If either is `nil`, the method returns early without calling `dismiss()`. The sheet has `.interactiveDismissDisabled()` set, so the user is stuck — no swipe-to-dismiss, no close button.

This could happen if:
- `complete()` is called but the TimerViewModel's `endDate` was never set (e.g., a race where `complete()` is re-entered)
- Someone calls `resetAfterSave()` before the user taps a category (there's no button to do this, but future code could)

- **Action:** Add a fallback: if the dates are nil, still call `dismiss()` with a `print("Warning: completedStartDate/completedEndDate nil in saveSession")`. Alternatively, show an alert and offer a "Discard" option.

### 6. `TimerViewModel.complete()` may cancel AlarmKit alarm during foreground transition

**Importance:** Medium | **Urgency:** Later | **Cost:** Medium

Per `CLAUDE.md`: "Timer.scheduledTimer keeps firing for a brief runloop-fade-out window after the app moves to background. If a tick can reach an 'end' state that cancels external system services (notifications, AlarmKit alarms, etc.), gate the cancellation on `UIApplication.shared.applicationState == .active`."

The code does gate on `.active`:

```swift
if UIApplication.shared.applicationState == .active {
    cancelNotification()
}
```

However, `complete()` is also reachable from `recalculateOnForeground()`, which is called in response to `UIApplication.willEnterForegroundNotification`. At that point `applicationState` is `.inactive` (not yet `.active`). If `recalculateOnForeground` → `complete()` runs during the `.inactive` → `.active` transition, the alarm IS preserved (because `.inactive != .active`). So the current code is *conservative* (safer) — it only cancels when truly `.active`. This is fine, but worth noting as a subtle coupling between notification timing and state.

- **Action:** No change needed for correctness, but add a comment noting the intentional conservatism and the `.inactive` → `.active` transition window.

### 7. iOS deployment target mismatch in tests

**Importance:** Low | **Urgency:** Later | **Cost:** Small

`project.pbxproj`:

| Target | `IPHONEOS_DEPLOYMENT_TARGET` |
|--------|------------------------------|
| Pomodoro (main app) | 17.6 |
| PomodoroWidgetExtensionExtension | 17.6 |
| PomodoroTests | **26.4** |

The test target requires iOS 26.4, but the app and widget target 17.6. This means tests can't run on simulators between 17.6 and 26.3 — they require a bleeding-edge runtime. This may be intentional (AlarmKit tests need 26.1+, and 26.4 is Xcode 26.4.1's minimum), but it means CI can't catch regressions on the main deployment target.

- **Action:** Set tests to 17.6, same as the app. Use `#available(iOS 26.1, *)` guards in test code for AlarmKit-specific tests. (Currently none of the tests exercise AlarmKit paths directly, so a lower target should work.)

### 13. `TimerViewModel` holds a persistent `CHHapticEngine` that may outlive the audio session

**Importance:** Low | **Urgency:** Later | **Cost:** Small

`ensureHapticEngine()` creates a `CHHapticEngine` and stores it in `hapticEngine`. The engine's `resetHandler` and `stoppedHandler` call `try? engine?.start()`. If the engine stops due to audio session interruption (e.g., phone call), the handler auto-restarts it. This is good, but on repeated interruptions the engine may cycle rapidly. Xcode console logs may show `CHHapticEngine` warnings (`Player start failed`, `Startup timeout`) during this — per CLAUDE.md these are accepted as noise.

- **Action:** No change needed. Documented as a known framework noise source.
